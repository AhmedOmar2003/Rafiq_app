RAFIQ Architecture Audit
Date: 2026-06-02
Auditor: Codex

Purpose
This document is a full architecture and launch-readiness audit for the RAFIQ
project in its current repository state. The goal is not to change code, but
to describe the actual system as-built:

- what is structurally solid
- what is functionally working
- what is still incomplete
- what is risky before production launch
- what should be done next, in priority order

This audit is intentionally direct. It does not assume that a feature is
"production ready" just because a UI exists or because a flow passes locally.

===============================================================================
2. Executive Summary
===============================================================================

Current judgment:
READY FOR BETA, NOT FULLY READY FOR HARD PRODUCTION

Why this is the verdict
- The system is much more coherent than before.
- Supabase is now the real backend center of gravity.
- The provider flow, real analytics flow, and campaign moderation flow are
  significantly more real than a mock/demo UI.
- Admin moderation surfaces exist and are wired to real data.

However, several launch-critical architectural gaps remain:
- subscription and billing are still demo-driven, not true payment lifecycle
- Sentry is documented but not active in the app runtime
- analytics integrity is not fully hardened against abuse
- admin schema source-of-truth is split between migrations and a manual SQL file
- staging and automated integration coverage are still insufficient
- storage cleanup is incomplete across all deletion paths

Bottom line
- The product is not "fake" anymore.
- It is functional and increasingly credible.
- But it is not yet a fully hardened production system for paid growth at scale.

3. System Architecture Overview
===============================================================================

3.1 Main system components

1. Flutter Mobile App
- End-user application for regular users and providers
- Handles auth, role selection, place browsing, place details, favorites,
  provider flow, subscriptions UI, provider hub, analytics surfaces, and
  campaign creation UI

2. Supabase Backend
- Primary operational source of truth for:
  - auth
  - profiles
  - roles
  - providers
  - places
  - subscriptions
  - reviews
  - favorites
  - moderation reports
  - appeals
  - analytics events
  - promotional campaigns

3. Admin Dashboard
- Next.js server-rendered back office
- Protected behind Supabase auth + admin role checks
- Uses service-role server client for moderation and admin operations

4. Provider Hub
- Provider-facing dashboard inside the Flutter app
- Loads provider entitlement, provider places, live analytics, and campaign
  status

5. Supabase Storage
- Buckets for place images, provider documents, banners, and campaign assets
- Bucket RLS is defined in SQL migrations

6. Analytics
- Raw event ingestion via analytics_events
- Historical aggregates via analytics_daily_rollups
- Live provider reads via provider_place_analytics_live RPC

7. Campaigns / Ads
- Provider creates place-linked campaigns
- Campaign is held in pending_review
- Admin dashboard approves or rejects
- Only active approved campaigns appear to end users

8. Authentication
- Supabase Auth is the active auth backbone
- Account mode and role state are reconciled against backend state

3.2 How components communicate

Flutter -> Supabase
- supabase_flutter client
- direct table reads/writes
- RPC calls for guarded business actions
- Edge Function invocation for Gemini chat

Admin Dashboard -> Supabase
- SSR client for session/user
- service-role client for admin operations and broad reads

Supabase -> Flutter/Admin
- direct PostgREST reads
- RLS-governed visibility
- real-time place updates in Provider Hub

3.3 Primary source of truth

The primary source of truth is now Supabase for most business entities.

This is true for:
- auth state
- profile/account mode
- provider rows
- places
- subscriptions rows
- favorites
- analytics events
- reports
- appeals
- promotional campaigns

However, there are still secondary local state layers:
- SharedPreferences for role state caching
- SharedPreferences for auth identity caching
- SharedPreferences for demo entitlement caching
- local device profile image storage

This means the architecture is improved, but not yet fully single-source.

3.4 Is there more than one backend truth?

Mostly no for current active business flows.

Important nuance:
- I did not find active PHP endpoint calls in the current production flows
  reviewed here.
- This is a major improvement over older architecture.
- But there are still legacy or dead remnants:
  - old networking packages
  - legacy helper code
  - stale assumptions in docs

So the current state is:
- active truth = Supabase
- residual technical debt = dead legacy artifacts, not active core flows

Strong areas

Potentially confusing areas

5.3 Source of truth concern: admin_roles drift

Important finding
- The dashboard depends on public.admin_roles
- But that table is not created inside the main Supabase migration history
- It exists in:
  /D:/rafiq_master/admin-dashboard-rafiq-app/supabase_admin_setup.sql

Why this matters
- The repository has two schema truth sources:
  1. supabase/migrations/
  2. admin-dashboard-rafiq-app/supabase_admin_setup.sql

This is an architectural smell.

Impact
- New environments can drift
- A fresh database restored only from migrations may miss admin_roles entirely
- Security and access assumptions become setup-script dependent

Severity
- High

5.4 RLS posture

General assessment
- The project has a mature RLS mindset
- Key tables are protected
- Public read rules are intentionally narrow
- role escalation is mostly prevented through user_roles design

Strong areas
- profiles not public
- user_roles write-blocked for users
- public places limited to approved + not deleted
- reviews limited to approved places
- provider document access is private and ownership-aware
- campaign creation is server-validated through SECURITY DEFINER RPC

Weak areas
- Some docs are stale and still describe favorites as disabled, while 0041
  re-enabled them
- admin_roles security model is outside main migrations, so the auditability
  of that surface is weaker than the rest of the backend

12. Security Audit
===============================================================================

12.1 Strong security areas

- RLS is a real architectural pillar, not an afterthought
- user_roles write path is protected
- provider campaign creation is backend-checked
- Gemini API key no longer lives in Flutter client code
- admin dashboard requires auth and admin role
- login throttling exists for admin login

12.2 Critical security concerns

CRITICAL-01: Analytics can be inflated by malicious clients

Evidence
- insert_event_batch is executable by anon and authenticated
- client sends session_id and dimensions
- server does not appear to validate place/provider consistency or apply
  server-side per-user rate limiting inside insert_event_batch

Impact
- place analytics can be artificially inflated
- provider trust in analytics can be undermined
- admin interpretation of place performance can be distorted

Recommendation
- move analytics ingestion to a stricter contract
- at minimum:
  - require authenticated writes for core analytics
  - rate-limit by user/session/IP
  - validate that provider_id matches place.provider_id when both are sent
  - optionally dedupe place_open by session/place/time window server-side

CRITICAL-02: Campaign impressions/clicks are incrementable by public callers

Evidence
- record_campaign_metric is granted to anon and authenticated
- it increments counters directly if campaign is active
- no dedupe, no server-side rate limiting, no signed one-time token, no
  session uniqueness

Impact
- campaign metrics are easy to game
- provider campaign ROI becomes untrustworthy

Recommendation
- do not increment counters directly from public callers without protection
- record raw click/impression events instead
- dedupe by session/IP/user/time window
- roll up metrics server-side

12.3 High security concerns

HIGH-01: Sentry is not active
- not a direct exploit
- but incident detection is weaker than ops docs imply

HIGH-02: admin_roles infrastructure drift
- schema not in main migrations
- environment parity risk

HIGH-03: delete cleanup is incomplete
- data erasure and storage hygiene are not fully consistent

HIGH-04: Gmail-only auth policy
- not a security exploit, but a strong product gate that can cause operational
  confusion if undocumented

12.4 Medium security concerns

MEDIUM-01: server actions often trust route-level guards
- acceptable internally
- but stronger per-action authorization would be better

MEDIUM-02: dead dependencies remain
- unnecessary packages increase audit surface and maintenance cost

12.5 API keys and secrets

Current state
- Gemini key is server-side
- service-role usage is server-side
- I did not find active service-role leakage in Flutter client code

Positive assessment
- This area is materially better than earlier architecture
